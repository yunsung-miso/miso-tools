# Loop Delegation — 루프 원칙 기반 업무 위임

## 핵심 원리

좋은 위임의 본질은 "무엇을 하라"가 아니라 "언제 끝나는가"를 정의하는 것이다.
루프 엔지니어링의 정의를 빌리면: 루프란 **정지 조건이 충족될 때까지 작업 사이클을 반복하는 것**이다.
위임도 마찬가지다. 정지 조건이 모호한 지시("깔끔하게 정리해줘")는 수신자가 언제 멈춰야 할지 몰라
과잉 작업하거나 미완성으로 끝난다. 정지 조건이 검증 가능한 지시는 수신자가 스스로 수렴한다.

이 스킬이 만드는 산출물은 다섯 가지 요소로 좋은 루프를 설계한다:
**명확한 목표, 유용한 컨텍스트, 작은 행동 단위, 신뢰할 만한 검증, 명시적인 정지 규칙.**

## 작업 순서

### 1단계: 입력 파악

쓰기 전에 세 가지를 확인한다. 대화에서 이미 알 수 있으면 묻지 말고 추론한다.

- **작업**: 무엇을 + 왜 (배경이 없으면 수신자는 잘못된 방향으로 최적화한다)
- **수신자**: 누가 받는가 — 아래 세 갈래 중 하나. 이게 산출물의 *형태*를 결정한다.
- **완료의 모습**: 끝났다는 걸 무엇으로 확인하는가

세 갈래 수신자:

| 수신자 | 산출물 형태 | 예 |
|---|---|---|
| **A. 사람 팀원** | 한국어 마크다운 지시서 파일 (그대로 전달) | 주니어에게 버그 수정, 디자이너에게 시안 요청 |
| **B. 인세션 서브에이전트** | `Agent` 도구에 그대로 넣는 프롬프트 문자열 | 지금 이 세션에서 서브에이전트로 병렬 처리 |
| **C. 독립·예약 에이전트** | 실행 설정(트리거·범위) + 프롬프트 | 매시간 CI 고치는 cron 에이전트, 상시 트리아지 |

세 번째(완료의 모습)가 비어 있으면 물어라. 이것 없이 쓴 지시서는 루프가 아니라 소원이다.
단, 맥락에서 합리적으로 유추 가능하면 유추한 기준을 쓰고 사용자가 검토하게 하는 편이
질문 공세보다 낫다.

### 2단계: 위임 유형 + Claude Code 실행 수단 선택

위임 사다리 — "사람이 무엇을 넘기는가" — 로 유형을 고른다. 위로 갈수록 지시서는 가벼워지고,
아래로 갈수록 가드가 무거워진다. 수신자가 B/C(에이전트)면 유형은 그대로 Claude Code의 실행
수단에 대응한다 — 지시서에 그 수단을 **명시**한다("매시간 실행"이 아니라 "`/schedule` cron").

| 유형 | 사람이 넘기는 것 | 언제 | Claude Code 실행 수단 | 산출물 특징 |
|---|---|---|---|---|
| **단발형** | 검증만 | 1회성·범위 명확 | `Agent` 단일 디스패치 (또는 그냥 직접) | 경량 — 목표·완료 기준·마감 |
| **수렴형** | 정지 조건 | 완료 기준은 검증 가능하나 몇 번의 시도가 필요 | `Workflow`(loop-until-dry / budget 루프) 또는 반복 상한 건 서브에이전트, `ralph`/`ultrawork` | 반복 상한·무진전 중단 필수 |
| **주기형** | 트리거 | 반복 발생하는 작업 스트림 | `/schedule` cron 에이전트 (인세션 폴링이면 `/loop`) | 트리거·1회 범위·소진 조건 |
| **상시형** | 일감 발견 | 스스로 일을 찾아 처리하는 루틴 | `/schedule` 자율 에이전트 | 가드 최엄격 — 범위 경계·에스컬레이션·중단 스위치 |

**과잉 설계 금지.** 모든 작업에 루프 구조가 필요한 건 아니다. "버튼 색 바꿔줘" 수준의 단발형에
7개 섹션 지시서를 쓰면 수신자는 지시서 읽는 데 작업보다 오래 걸린다. 단발형은 목표 + 완료 기준 +
컨텍스트 링크 + 마감, 4줄이면 충분할 수 있다. 복잡한 구조는 수렴형 이상에서만 꺼낸다.

### 3단계: 산출물 작성 (수신자 형태별)

작업에 맞지 않는 섹션은 뺀다. 빈 섹션을 형식적으로 채우는 것이 가장 나쁜 결과다.

#### A. 사람 팀원 — 마크다운 지시서

```markdown
# [작업 제목]

## 목표
[한두 문장. 무엇을 + 왜. "왜"가 있어야 수신자가 예외 상황에서 올바르게 판단한다]

## 완료 기준 (이게 전부 참이면 끝)
- [ ] [검증 가능한 조건 — 실행 가능한 명령, 확인 가능한 상태, 측정 가능한 수치]
[나쁜 예: "코드가 깨끗해야 함" / 좋은 예: "pnpm test 전체 통과, 신규 lint 경고 0건"]

## 컨텍스트
[관련 파일·링크·이전 시도·제약. 수신자가 되물을 것을 미리 답한다]

## 작업 방식
[권장 순서와 작업 단위. 작게 쪼개 각 단계를 확인 가능하게.
시니어면 방식은 간략히 — 목표·기준만으로 충분할 수 있다]

## 검증
[누가, 어떻게 확인하는가. 본인의 "다 됐어요"는 검증이 아니다 — maker/checker 분리.
리뷰어를 이름으로 지정, 셀프 머지 금지]

## 가드
- 무진전 중단: [예: 반나절 원인 못 좁히면 그 시점까지 정리해 공유 — 정상 절차]
- 에스컬레이션: [범위 확대·다른 시스템 영향·예상 밖 발견이면 즉시 알릴 것]

## 기록
[시도·통과·막힌 것을 어디에 남길지 — PR 설명, Linear 코멘트. 이 기록이 다음 위임을 개선한다]
```

#### B. 인세션 서브에이전트 — `Agent` 프롬프트

지금 세션에서 `Agent` 도구(또는 `Task`)에 그대로 넣을 **프롬프트 문자열 + 디스패치 설정**을 낸다.
사람용 산문을 서브에이전트에게 주지 않는다 — 서브에이전트는 기계 검증 기준과 숫자 가드에 반응한다.

```markdown
**subagent_type**: [general-purpose / code-reviewer / Explore / executor 등]
**run_in_background**: [병렬이면 true]

--- 프롬프트 ---
[목표 1문장.]

완료 기준(전부 충족 시 종료):
- [기계 검증 가능: 실행할 명령어 + 기대 결과. 예: `pnpm lint` 경고 0, `gh pr checks 123` 실패 0]

컨텍스트: [파일 경로·제약. 서브에이전트는 신선한 컨텍스트다 — 필요한 건 전부 넣는다]

가드:
- 최대 [N]회 반복. 2회 연속 diff 무변화면 중단하고 현 상태 보고.
- [금지 사항: 예 — 테스트 삭제/skip 금지, main 직접 push 금지]

반환: [최종 메시지로 무엇을 돌려줄지. 구조화가 필요하면 형식 지정]
```

검증은 **별도** 디스패치로 — 작업한 에이전트에게 자기 검증을 맡기지 않는다(4단계 참조).

#### C. 독립·예약 에이전트 — 실행 설정 + 프롬프트

`/schedule`(cron) 또는 `/loop`로 돌릴 것. 실행 설정과 프롬프트를 분리해 낸다.

```markdown
**실행**: /schedule [cron 또는 자연어 주기] — 예: 매시간 정각
**1회 실행 범위**: [그 시점에 처리할 대상 전부. 수렴형 목표를 실행 안에 내장]
**소진 조건**: [처리할 게 없으면 "이상 없음"만 기록하고 즉시 종료 — 일을 만들어내지 않는다]

--- 프롬프트 ---
[B와 동일 구조 + 아래]

상태 기록: [상태 파일 위치. 다음 실행이 이전이 멈춘 곳에서 이어받게 — 중복 시도 방지]
누적 가드: [여러 실행에 걸친 상한. 예: 같은 대상 누적 5회 시도 후 사람 개입 요청 코멘트만]
```

### 4단계: 검증·가드에 실제 도구 연결

추상적 "검증"·"기록"을 수신자가 실제로 쓰는 도구로 못박는다. maker와 checker는 항상 분리한다 —
컨텍스트가 신선한 검증자가 자기 검증보다 훨씬 신뢰할 만하다.

- **checker 분리**: 서브에이전트 작업이면 리뷰는 별도 `Agent`(`code-reviewer` 등) 또는
  `superpowers:requesting-code-review` / `verification-before-completion`. 사람이면 리뷰어를 이름으로.
- **gh**: 완료 기준·검증에 `gh pr checks <n>`(실패 0), 로그는 `gh run view <id> --log-failed`.
  사람용 기록은 PR 코멘트, 수정 커밋마다 "검토 후 문제 있으면 revert" 코멘트.
- **Linear**: 작업 기록·핸드오프는 이슈/코멘트. 위임 원본이 티켓이면 완료 시 상태 이동을 기준에 포함.
- **Datadog**: 상시·주기형의 정지/건강 조건에 쿼리·모니터를 건다(예: 에러율 임계 이하 확인).
- **git 워크플로**: 가드로 — main 직접 push 금지, 브랜치에서 작업, PR로만 병합, atomic 커밋.
- **Figma**: 디자인 산출물 검증은 스크린샷/variable 확인을 완료 기준에 넣는다.

**에이전트 가드는 숫자로**: 최대 턴/반복, diff 무변화 감지 시 중단, 비용/토큰 상한.
상한 없는 무한 루프는 루프가 아니라 열린 청구서다.

**사람 가드는 시간으로**: 타임박스, "N시간 막히면 물어보기". 반복 상한·토큰 예산 같은 기계 용어는
쓰지 않는다. 에스컬레이션은 실패 보고가 아니라 정상 절차임을 명시한다.

### 5단계: 산출과 마무리

- 사람용(A)은 마크다운 파일로 저장, 파일명 `지시서-{작업요약}.md`. B/C는 그대로 실행 가능한 블록으로.
- 전달할 때 **사용자 확인이 필요한 판단 지점**(유추한 완료 기준, 임의로 정한 상한·마감,
  고른 실행 수단)을 2~3줄로 짚어준다. 나머지 설명은 생략 — 산출물이 스스로 말하게 한다.

## 예시

**예시 1 — 수렴형, 사람(주니어)**
입력: "주니어한테 검색 무한스크롤 중복 아이템 버그 맡기려고"
산출(핵심): 완료 기준 "같은 검색어 5페이지 스크롤 시 중복 0건 / 기존 E2E 전체 통과 / 재발 방지 테스트 1건" ·
검증 "PR 올리면 OO 리뷰, 셀프 머지 금지" · 가드 "반나절 원인 못 좁히면 정리해 공유 — 정상 절차" ·
기록 "PR 설명에 원인·수정·버린 접근".

**예시 2 — 인세션 서브에이전트(B)**
입력: "이 세션에서 서브에이전트로 4개 모듈 lint 에러 병렬로 고쳐줘"
산출(핵심): `subagent_type: executor`, `run_in_background: true` 4개 디스패치 · 완료 기준
"`pnpm lint <모듈>` 경고 0" · 가드 "최대 3회, 무변화 시 중단·보고, 다른 파일 건드리지 말 것" ·
검증은 별도 `code-reviewer` 디스패치로 4개 diff 일괄 리뷰.

**예시 3 — 주기형+수렴형, 예약 에이전트(C)**
입력: "claude한테 매시간 miso-native 열린 PR CI 실패 고치게"
산출(핵심): 실행 "`/schedule` 매시간" · 1회 범위 "그 시점 CI `failure` PR 전부(draft 제외)" ·
소진 조건 "실패 PR 없으면 상태 파일에 '이상 없음' 기록 후 종료" · 완료 기준 "`gh pr checks` 실패 0
또는 원인 분석 코멘트 게시" · 가드 "PR당 3회, 누적 5회 후 사람 개입 코멘트, 테스트 삭제·workflow 수정·
main push 금지" · 검증 "수리 커밋 머지 금지, 사람 리뷰" · 기록 "상태 파일 + PR 코멘트".

**예시 4 — 단발형, 과잉 설계 회피**
입력: "디자이너한테 앱 아이콘 시안 3개 요청, 다음 주 화까지"
산출: 4~6줄 경량 지시서 (목표·수량·참고 자료·마감·전달 형식). 가드·기록 섹션 없음.
